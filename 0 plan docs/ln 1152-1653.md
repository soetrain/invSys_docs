## Root Cause: Confirmed

After reading the full [`cDynItemSearch.cls`](https://github.com/justinwj/invSys_fork/blob/cbf3e6b18563cadaacfb0144d296e11df5cafbe5/src/Core/ClassModules/cDynItemSearch.cls), the bug is clearly located and the failure mode is exact.

### The Real Problem — Two Converging Flaws

**Flaw 1: `ResolveManagedInventoryTable()` uses `ResolveSearchWorkbook()`, which returns the *calling* workbook — which is ShippingBOM.xlsb.**

```vba
Private Function ResolveManagedInventoryTable() As ListObject
    Dim wb As Workbook
    Set wb = ResolveSearchWorkbook()        ' ← Returns callbackCell.Parent.Parent
    If Not wb Is Nothing Then
        Set ResolveManagedInventoryTable = FindInventoryTableInWorkbook(wb)
        If Not ResolveManagedInventoryTable Is Nothing Then Exit Function
    End If
    ' ... fallback scans ALL open workbooks
End Function
```

`ResolveSearchWorkbook()` returns the workbook containing the `callbackCell`. When the user clicks a `BoxBOM` cell, the `callbackCell` lives in **ShippingBOM.xlsb** (or whatever the Shipping workbook is called). So `wb` = the Shipping workbook — not the inventory workbook.

**Flaw 2: `FindInventoryTableInWorkbook()` only looks for a sheet named `"inventorymanagement"` etc. — but if ShippingBOM.xlsb has no such sheet, the local search fails and the fallback scans ALL open workbooks, picking up the first one with an `invSys` table it can find — which could be a stale/polluted local read-model containing box/package rows.**

```vba
Private Function FindInventoryWorksheetInWorkbook(ByVal wb As Workbook) As Worksheet
    For Each ws In wb.Worksheets
        Select Case LCase$(Trim$(ws.Name))
            Case "inventorymanagement", "inventory management", "inventory_management", "inventory"
                Set FindInventoryWorksheetInWorkbook = ws
                Exit Function
        End Select
    Next ws
End Function
```

**Flaw 3: `LoadShippingComponentItems` falls back to `LoadManagedInventoryItemsLocal()`, which calls `ResolveManagedInventoryTable()` again — the same poisoned resolver.** There is no path that enforces a strict inventory-only source before handing data to the picker.

### Why Box/Package Rows Appear

The local read-model on the Shipping workbook (populated by `modOperatorReadModel.RefreshInventoryReadModelForWorkbook`) may contain rows sourced from ShippingBOM package data if the snapshot was stale, was written by the wrong publisher, or if the `invSys` table in that workbook was overwritten with package-origin rows during a prior broken refresh. The `InventoryPickerTableHasRows()` guard only checks for `ROW` and `ITEM` columns — both of which exist in ShippingBOM package tables (e.g., `T11box`, `T10box` have ROW keys 79, 78, 72). So validation passes and box rows load.

## Precise Fix

The changes are all in `cDynItemSearch.cls`, concentrated in three functions. No other files need changes for the core fix.

### 1. Add a strict table-name blocklist to `FindInventoryTableInWorkbook`

```vba
Private Function FindInventoryTableInWorkbook(ByVal wb As Workbook) As ListObject
    Dim ws As Worksheet
    Dim lo As ListObject

    Set ws = FindInventoryWorksheetInWorkbook(wb)
    If ws Is Nothing Then Exit Function

    On Error Resume Next
    Set lo = ws.ListObjects("invSys")
    On Error GoTo 0
    If lo Is Nothing Then Exit Function

    ' STRICT CONTRACT: reject if workbook path smells like ShippingBOM domain
    If IsShippingDomainWorkbook(wb) Then
        Debug.Print "FindInventoryTableInWorkbook: REJECTED wb '" & wb.Name & "' — shipping domain source"
        Set FindInventoryTableInWorkbook = Nothing
        Exit Function
    End If

    ' Extra column guard for shipping_component context
    Set FindInventoryTableInWorkbook = lo
End Function

Private Function IsShippingDomainWorkbook(ByVal wb As Workbook) As Boolean
    Dim wbLower As String
    wbLower = LCase$(Trim$(wb.Name))
    ' Reject if workbook name implies it is a ShippingBOM/package workbook
    Dim rejectTerms As Variant
    rejectTerms = Array("shippingbom", "shipping_bom", "boxbom", "boxbomversions", _
                        "aggregateboxbom", "shipmentstally", "shippingbomview")
    Dim i As Long
    For i = LBound(rejectTerms) To UBound(rejectTerms)
        If InStr(wbLower, CStr(rejectTerms(i))) > 0 Then
            IsShippingDomainWorkbook = True
            Exit Function
        End If
    Next i
End Function
```

## Performance Root Causes

There are four distinct problems causing the blinking and slowness, ordered by impact.

### 1. `Workbooks.Open` on Every Box Selection (The Biggest Offender)

Every time the user selects a box, the system is calling `Workbooks.Open` on inventory/inbox/auth workbooks that are **already open** in the session. In `modInventoryDomainBridge.bas`, `modRoleEventWriter.bas`, `modWarehouseSync.bas`, and `modInventoryApply.bas`, the pattern is:

```vba
' Current bad pattern:
Set wb = Application.Workbooks.Open(targetPath)   ' ← opens EVEN IF already open
```

When Excel opens a workbook that is already open, it either: (a) flashes the workbook's window, causing visible blink, or (b) triggers a "file modified" dialog that `DisplayAlerts = False` eats silently, then re-opens it anyway. Either way = blink + overhead.

The `modRoleEventWriter.bas` already sets `Application.ScreenUpdating = False` before opening, but that only suppresses one layer of the flash.

**Fix:** Add an "already open?" guard before every `Workbooks.Open` call:

```vba
' Shared helper — add to modRuntimeWorkbooks.bas
Public Function GetOrOpenWorkbook(ByVal filePath As String, _
                                  Optional ByVal readOnly As Boolean = False) As Workbook
    Dim normalPath As String
    normalPath = NormalizePath(filePath)         ' lowercase, canonical separators

    ' Check if already open — zero I/O cost
    Dim wb As Workbook
    For Each wb In Application.Workbooks
        If StrComp(NormalizePath(wb.FullName), normalPath, vbTextCompare) = 0 Then
            Set GetOrOpenWorkbook = wb
            Exit Function
        End If
    Next wb

    ' Only opens if not already in the collection
    Dim prev As Long
    prev = Application.AutomationSecurity
    Application.AutomationSecurity = msoAutomationSecurityForceDisable
    Set GetOrOpenWorkbook = Application.Workbooks.Open( _
        Filename:=filePath, UpdateLinks:=0, ReadOnly:=readOnly, _
        Notify:=False, AddToMru:=False)
    Application.AutomationSecurity = prev
End Function
```

Then replace every bare `Workbooks.Open` in `modInventoryDomainBridge`, `modRoleEventWriter`, `modWarehouseSync`, and `modInventoryApply` with `GetOrOpenWorkbook(path)`.

### 2. Missing `modUiQuiet` Wrapper Around BoxBOM Save Path

`modTS_Received.bas` uses `modUiQuiet.BeginQuietUi` / `EndQuietUi` properly.  The BoxBOM save path almost certainly does not, because it was written after those guards existed. Every cell write, table row addition, and `ListObject` operation during a box save triggers a recalculation + repaint unless wrapped.

**Fix:** Wrap the entire BoxBOM commit operation:

```vba
    Set wb = callbackCell.Parent.Parent

    modUiQuiet.BeginQuietUi wb         ' ← ScreenUpdating=F, Calc=Manual, Events=F
    On Error GoTo CleanupQuiet

    '--- all BoxBOM writes here ---

CleanupQuiet:
    modUiQuiet.EndQuietUi wb           ' ← always restores, even on error
    If Err.Number <> 0 Then Err.Raise Err.Number
End Sub
```

If `modUiQuiet` is not yet used in the Shipping module, add a direct guard at minimum:

```vba
Dim prevScreen As Boolean
Dim prevCalc As XlCalculation

prevScreen = Application.ScreenUpdating

Application.ScreenUpdating = False
Application.Calculation    = xlCalculationManual

On Error GoTo RestoreAppState

RestoreAppState:

Application.Calculation    = prevCalc

```

### 3. `TryRefreshSearchInventoryReadModel` Triggers a Full Workbook Surface Rebuild on Picker Open

In `cDynItemSearch.LoadManagedInventoryItems`, if the table is empty it calls `TryRefreshSearchInventoryReadModel()`, which calls:


That surface rebuild writes cells, possibly adds sheets, and fires recalculations — all during picker open. This explains the "a lot of time between box selections."

**Fix:** Decouple the refresh from the picker. The picker should **never trigger a background refresh** in the hot path. Refresh should be explicit (a ribbon button or a timed background event):

```vba

Private Function LoadManagedInventoryItems(...) As Variant
    Dim loInv As ListObject
    Set loInv = ResolveManagedInventoryTable()
    If loInv Is Nothing Then Exit Function
    If loInv.DataBodyRange Is Nothing Then Exit Function

    LoadManagedInventoryItems = BuildInventoryPickerItemsFromTable(loInv, includeCategory)
End Function
```

If an auto-refresh is truly needed, move it to a non-blocking `Application.OnTime` call that fires after the picker is already visible:

```vba
' Fire refresh 500ms after picker opens, non-blocking:
Application.OnTime Now + TimeValue("00:00:01"), "mBackground.RefreshInventoryReadModel"
```

## Short Answer

**Yes, a UserForm-centered architecture will be dramatically more stable and performant than the current ribbon-buttons-on-Tables approach** — for this specific use case. But the reason is not "UserForms are faster." The reason is that a UserForm forces a clean separation of concerns that the ribbon-on-sheet model inherently resists, and that separation is what eliminates the cascading event/refresh/open cycles that are killing you.

***

## Why the Current Model Keeps Regressing

The ribbon-on-Tables architecture creates a structural trap that no amount of patching fully escapes. Every fix introduces a new interaction surface.

**The cascade chain that fires on every box selection:**

```
User clicks cell in BoxBOM table
  → Worksheet_SelectionChange fires
    → cDynItemSearch.ShowForShippingComponentCell
      → LoadCanonicalManagedInventoryItems
        → modInventoryDomainBridge.ResolveInventoryWorkbookBridge
          → Workbooks.Open (if not cached)
            → Excel window flash
  → User picks item → CommitSelection
    → callbackCell.Value = x        → Worksheet_Change fires
    → callbackCell.ListObject write → Worksheet_Change fires again
      → version autofill logic       → more cell writes
        → Worksheet_Change fires again (re-entrant)
          → Application.EnableEvents = False needed
            → but someone forgot it somewhere
              → regression
```

`modTS_Shipments.bas` being 117 KB means this logic has grown organically inside one module, and the event handler / quiet-ui guard boundaries are now inconsistent.  Codex keeps "gassing out" because it is patching a system where every write can trigger another write — the state machine is implicit and lives in Excel's event system.

***

## The UserForm Model — What Changes Structurally

A UserForm does not replace ribbon buttons — you keep the ribbon. What changes is **where state lives and where writes happen**.

### Current model (broken loop):

```
Sheet ←→ Worksheet_Change ←→ modTS_Shipments ←→ Workbooks.Open
  ↑_____________writes_____________________________|
```


### UserForm model (clean pipeline):


# The ~5 second Make Boxes time is now almost certainly the inbox open/save path, not list selection. The picker improves navigation; getting posting under 1000 ms likely needs changing how the inbox write is persisted. What do you think the best course of action is?

The source confirms exactly what you suspected. Here's the precise analysis and the best course of action.

***

## What `QueueEventCore` Actually Does Per Click

Reading `QueueEventCore` in `modRoleEventWriter.bas` top to bottom, the latency budget for a single `Make Boxes` click on a NAS/UNC path is:


| Step | What happens | Typical cost on NAS |
| :-- | :-- | :-- |
| `EnsureContextResolved` | `modConfig.LoadConfig` | 10–50 ms |
| `modAuth.LoadAuth` | Opens auth workbook read-only, scans tables | 50–300 ms |
| `ResolveInboxWorkbookForEventType` | **Opens `invSys.Inbox.Shipping.*.xlsb` over UNC** | **1000–4000 ms** |
| `EnsureShipInboxSchema` | Schema validation pass | 10–50 ms |
| `lo.ListRows.Add` + 18 × `SetTableRowValueRole` | Cell writes | < 50 ms |
| `SaveWorkbookRole` → `wb.Save` | **Saves .xlsb to NAS** | **500–2000 ms** |
| `CloseTransientRoleWorkbook` | Close + possibly re-save | 200–500 ms |

The **two NAS I/O walls** are the workbook open and the `wb.Save`. Together they account for nearly all of the 5-second budget. Everything else is negligible.

There is also a structural inefficiency: every `QueueEventCore` call opens the inbox, writes one row, saves the full `.xlsb`, and closes it — all synchronously inside the button click. The `.xlsb` format is compact but Excel's COM save path is slow because it must recalculate and serialize the entire workbook object model each time.

***

## Best Course of Action

There are three realistic options. Here is an honest assessment of each:

### Option A — Keep-alive the inbox workbook (smallest change, fastest win)

Instead of open → write → save → close on every event, **keep the inbox workbook open in a hidden state** for the session and only save/close it when the add-in unloads or the user explicitly flushes.

```vba
' In modRoleEventWriter.bas — add a module-level cache
Private mInboxWorkbooks As Object   ' key: fullPath → Workbook

Private Function GetOrOpenInboxWorkbook(ByVal fullPath As String, ...) As Workbook
    If mInboxWorkbooks Is Nothing Then
        Set mInboxWorkbooks = CreateObject("Scripting.Dictionary")
        mInboxWorkbooks.CompareMode = vbTextCompare
    End If

    If mInboxWorkbooks.Exists(fullPath) Then
        Dim cached As Workbook
        Set cached = mInboxWorkbooks(fullPath)
        ' Validate it is still open and writable
        On Error Resume Next
        Dim testName As String
        testName = cached.Name
        On Error GoTo 0
        If testName <> "" And Not cached.ReadOnly Then
            Set GetOrOpenInboxWorkbook = cached
            Exit Function
        End If
        mInboxWorkbooks.Remove fullPath
    End If

    ' Normal open path, then cache it
    Dim wb As Workbook
    Set wb = ... ' existing open logic
    If Not wb Is Nothing Then mInboxWorkbooks(fullPath) = wb
    Set GetOrOpenInboxWorkbook = wb
End Function
```

Then change `SaveWorkbookRole` inside `QueueEventCore` to a **deferred save** — only flush to disk after a timeout or on workbook close:

```vba
' Replace: SaveWorkbookRole wbInbox
' With:    MarkInboxDirtyRole wbInbox  (just sets wb.Saved = False conceptually)
' Then flush on: App_WorkbookBeforeClose, a ribbon "Flush" button, or a 30-second OnTime timer
```

**Cost of open eliminated. Cost of save amortized across multiple events.**
Expected result: posting drops from ~5 s to ~50–100 ms per click. The tradeoff is that unsaved events sit in-memory if Excel crashes before flush — but the processor already handles re-queuing from the inbox, so a crash between write and flush only loses in-flight events, not already-processed ones.

***

### Option B — Write to a local staging file first, sync to NAS in background (more robust, more code)

Write the event row to a **local temp `.xlsb`** (fast, `< 50 ms`), return the `EventID` immediately, then push the local file to NAS on a background `OnTime` timer. The processor already reads from a known inbox path, so the sync just means copying or appending the local staging rows into the NAS inbox file.

This is the most resilient design for intermittent NAS connectivity but requires a staging-merge step that doesn't exist yet.

***

### Option C — Write a CSV/TSV append instead of a .xlsb save (breaks format contract)

Replace the entire `.xlsb` inbox with a flat append-only `.csv` or `.tsv` file. File append is ~5 ms even on NAS. But this would require rewriting `modProcessor` and breaks the existing schema contract — too disruptive for a targeted perf fix.

***

## Recommended Path: Option A, in two steps

**Step 1 — Eliminate the open cost** (immediate, ~1 day of work):

Change `ResolveInboxWorkbookForEventType` to return a cached open workbook when one exists for the session. The inbox workbook stays hidden and open after the first event write. No changes to processor, no format changes.

**Step 2 — Defer the save** (follow-up, ~half day):

Change `SaveWorkbookRole` inside `QueueEventCore` to a dirty-mark instead of an immediate save. Add a flush in `App_WorkbookBeforeClose` in `cAppEvents.cls` and optionally an `OnTime` timer every 60 seconds. This eliminates the NAS save latency from the click path entirely.

The result is that `QueueEventCore` only touches the NAS on first open per session and on periodic/close flushes — not once per button click. With the Receiving readiness fix already landing, **the combined effect should bring `Make Boxes` well under 200 ms** in the steady state (session already has the inbox open).

# ~5 ms is very fast compared to < 50 ms, seems like Option C would be worth implementing later?

Yes, but with important caveats that change the calculus. Here's the honest picture:

## Why ~5 ms Is Real, But Not the Full Story

The ~5 ms figure for a flat file append is accurate for a **local** file. On a NAS/UNC path that number realistically becomes **30–150 ms** depending on SMB negotiation overhead — still dramatically faster than `.xlsb` save, but not quite as magical as it sounds.

The more significant problem is what `modProcessor.RunBatch` reveals: **the processor is not a passive file reader**.

### What the Processor Actually Does With the Inbox

Looking at `RunBatch` and `UpdateInboxRowStatus`:

```vba
' After processing EACH row, the processor does:
SetCellByColumnProcessor lo, rowIndex, "Status", newStatus
' ... sets ErrorCode, ErrorMessage, FailedAtUTC ...
SaveWorkbookProcessor lo.Parent.Parent   ' ← full .xlsb save per row
```

The inbox is a **read-write contract** — the processor stamps `PROCESSED`, `SKIP_DUP`, or `POISON` back onto the same rows, saving after every single row. A flat CSV/TSV file cannot be randomly row-updated like this without rewriting the entire file, which eliminates the append speed advantage entirely on the processor side. You'd need either:

1. A **sidecar status file** (e.g. `*.status.csv` alongside the main inbox) that records processed EventIDs — the processor appends to it rather than updating rows in-place
2. Or the **two-file pattern**: an append-only write inbox + a separate processed/tombstone log the processor maintains

Both of those are significant architectural changes across `modProcessor`, `modRoleEventWriter`, and anything that calls `DescribeInboxPendingRows` (which also scans Status columns in-place).

## The Real Cost/Benefit

| Approach | Write latency | Processor changes | Risk |
| :-- | :-- | :-- | :-- |
| Option A (keep-alive + deferred save) | ~50–100 ms → ~5–20 ms steady state | Zero | Low |
| Option C (CSV append) | ~30–150 ms per write | Major — status tracking, schema, `EnsureInboxSchema`, `UpdateInboxRowStatus` | High |

Option C shaves maybe 30–80 ms off the write side compared to a well-implemented Option A — but the processor currently saves the inbox `.xlsb` after **every processed row**, meaning the dominant I/O cost has already shifted to the processor side anyway. Fixing that (batching processor saves to once per `RunBatch`) would likely have more impact than the write format.

## Recommended sequence

1. **Now**: Option A (keep-alive + deferred save) — fast, zero risk, gets you under 200 ms
2. **Follow-up**: Batch processor saves — change `UpdateInboxRowStatus` to only call `SaveWorkbookProcessor` once at the end of the row loop, not per row. That's a two-line change with large impact on processor throughput
3. **Later, if warranted**: Option C is worth revisiting only if you ever move to a high-volume write scenario (dozens of events per second), which the current warehouse scale probably doesn't need. At that point the whole inbox architecture would likely change anyway

# after release 1 of invSys, and probably some other developments, and fine tuning,R2 F\# is next target for performance. Would it make sense to implement Option C in this rebuild?

Yes — and the R2 context makes Option C not just sensible but the **natural fit**. Here's why the framing changes completely.

## The Key Insight: R2 Removes the Constraint That Made Option C Expensive

The entire reason Option C was risky in R1 is that `modProcessor` uses the inbox as a **mutable Excel table** — it stamps `PROCESSED`/`POISON` back onto individual rows and saves after each one. That in-place mutation is what forces the format to be a random-access workbook.

In R2, the processor moves to **F\# using ClosedXML** — running outside Excel automation entirely.  That removes the VBA constraint and lets you redesign the inbox contract from scratch. An F\# processor can trivially work with an append-only event log because it naturally handles immutable-record pipelines and can maintain its own processed-EventID index without touching the source file.[^9_1]

## What Option C Looks Like as a Native F\# Design

Instead of a mutable `.xlsb` table, the inbox becomes a proper **append-only event log**:

```
invSys.Inbox.Shipping.ST1.jsonl       ← writer appends one JSON line per event
invSys.Inbox.Shipping.ST1.processed   ← processor appends processed EventIDs here
invSys.Inbox.Shipping.ST1.poison      ← processor appends poison records here
```

The VBA writer side (`QueueEventCore`) just does:

```vba
' ~5-30 ms even on NAS — sequential append, no workbook open
Open fullPath For Append As #fileNum
Print #fileNum, BuildEventJson(...)
Close #fileNum
```

The F\# processor side reads the log, diffs against the processed index, applies events, and appends to the processed file — never mutating the source. This maps perfectly to F\# discriminated unions and pipelines:

```fsharp
type InboxEvent = { EventId: string; EventType: string; Payload: JsonElement; ... }
type ProcessResult = Applied | SkipDup | Poison of code: string * msg: string

let processInbox inboxPath processedPath =
    let processed = loadProcessedIds processedPath
    File.ReadLines(inboxPath)
    |> Seq.map parseEvent
    |> Seq.filter (fun e -> not (processed.Contains e.EventId))
    |> Seq.map (applyEvent inventoryWb)
    |> Seq.iter (appendResult processedPath)
```

This also solves the **processor save-per-row problem** identified earlier —  there's no workbook to save mid-loop at all.

## One Caveat: The VBA Inbox Reader

The R2 plan still has VBA on the station side (Excel is the UI). `DescribeInboxPendingRows` and `OpenInboxWorkbook` in `modRoleEventWriter` need updated readers that parse `.jsonl` instead of opening a workbook. That's moderate work but straightforward — a sequential line-scan with `Line Input` is fast and has no COM overhead.

## Verdict

|  | R1 Option A | R2 Option C (.jsonl) |
| :-- | :-- | :-- |
| Write latency | ~20–50 ms (keep-alive) | ~5–30 ms |
| Processor complexity | Same VBA pattern | Simpler in F\# (immutable pipeline) |
| Status tracking | In-place row mutation | Sidecar append file |
| F\#→Rust portability | N/A | Excellent — file I/O maps 1:1 |
| Implementation risk | Low (VBA tweak) | Low in F\# (high in VBA) |

Do Option A now to unblock R1 performance. Design the R2 F\# processor with `.jsonl` append-only inboxes from the start — don't try to keep the `.xlsb` inbox format and bolt F\# on top of it. The format change *is* the rebuild, and F\# is the right tool to implement it cleanly.[^9_1]

# resilience is important, how would option A + option B path work?

Good question — they're actually complementary layers, not alternatives. Here's how they stack:

## The Two-Layer Model

**Option A** (keep-alive + deferred save) eliminates latency from the *happy path*. **Option B** (local staging → NAS sync) eliminates the *failure mode* where NAS is unreachable at the moment of posting. Together they form a write-ahead pattern that is standard in any durable queue design.

```
[User clicks Make Boxes]
        │
        ▼
┌─────────────────────────┐
│  1. Append to LOCAL     │  ← ~5 ms, always succeeds
│     staging file        │     even if NAS is down
└────────────┬────────────┘
             │ return EventID to caller immediately
             ▼
┌─────────────────────────┐
│  2. Mark local file     │  ← Option A: in-memory wb
│     dirty; don't save   │     kept open, no NAS touch
│     to NAS yet          │     on this click
└────────────┬────────────┘
             │ background / on-timer / on-close
             ▼
┌─────────────────────────┐
│  3. Sync local → NAS    │  ← Option B: merge staged
│     inbox               │     rows into NAS inbox wb,
└─────────────────────────┘     then save once
```

The key is that step 1 always succeeds — the click returns immediately regardless of NAS state.

## Concrete Implementation

### Layer 1 — Local staging file (Option B write side)

```vba
' modRoleEventWriter.bas — new private function
Private Function AppendToLocalStagingInbox(ByVal eventType As String, _
                                            ByVal resolvedWh As String, _
                                            ByVal resolvedSt As String, _
                                            ByVal rowDict As Object, _
                                            ByRef errorMessage As String) As Boolean
    Dim stagingPath As String
    Dim fileNum As Integer

    stagingPath = LocalStagingPathRole(eventType, resolvedSt)
    If stagingPath = "" Then
        errorMessage = "Could not resolve local staging path."
        Exit Function
    End If

    fileNum = FreeFile
    On Error GoTo FailAppend
    Open stagingPath For Append As #fileNum
    Print #fileNum, DictionaryToJsonRole(rowDict)   ' one JSON line per event
    Close #fileNum
    AppendToLocalStagingInbox = True
    Exit Function

FailAppend:
    errorMessage = "Local staging append failed: " & Err.Description
    On Error Resume Next
    Close #fileNum
End Function

Private Function LocalStagingPathRole(ByVal eventType As String, _
                                       ByVal stationId As String) As String
    ' Always local — Environ("LOCALAPPDATA") or ThisWorkbook.Path
    Dim localRoot As String
    localRoot = Environ$("LOCALAPPDATA")
    If localRoot = "" Then localRoot = Environ$("TEMP")
    LocalStagingPathRole = localRoot & "\invSys\staging\" & _
                           InboxWorkbookNameRole(eventType, stationId) & ".staging.jsonl"
End Function
```


### Layer 2 — NAS inbox keep-alive (Option A)

```vba
' Module-level cache — inbox workbook stays open for the session
Private mInboxCache As Object   ' Dictionary: fullPath → Workbook
Private mInboxDirty As Object   ' Dictionary: fullPath → Boolean

Private Function GetCachedInboxWorkbook(ByVal fullPath As String, _
                                         ByRef errorMessage As String) As Workbook
    If mInboxCache Is Nothing Then
        Set mInboxCache = CreateObject("Scripting.Dictionary")
        mInboxCache.CompareMode = vbTextCompare
        Set mInboxDirty = CreateObject("Scripting.Dictionary")
        mInboxDirty.CompareMode = vbTextCompare
    End If

    If mInboxCache.Exists(fullPath) Then
        Dim wb As Workbook
        Set wb = mInboxCache(fullPath)
        On Error Resume Next
        Dim nameCheck As String
        nameCheck = wb.Name     ' will error if workbook was closed externally
        On Error GoTo 0
        If nameCheck <> "" And Not wb.ReadOnly Then
            Set GetCachedInboxWorkbook = wb
            Exit Function
        End If
        mInboxCache.Remove fullPath   ' stale — drop and re-open
    End If

    Set GetCachedInboxWorkbook = ResolveInboxWorkbookForEventType(...)
    If Not GetCachedInboxWorkbook Is Nothing Then
        mInboxCache(fullPath) = GetCachedInboxWorkbook
        mInboxDirty(fullPath) = False
    End If
End Function

Public Sub FlushInboxCache()
    ' Called by: App_WorkbookBeforeClose, ribbon "Flush", OnTime timer
    If mInboxCache Is Nothing Then Exit Sub
    Dim key As Variant
    For Each key In mInboxCache.Keys
        If mInboxDirty.Exists(CStr(key)) Then
            If CBool(mInboxDirty(CStr(key))) Then
                SaveWorkbookRole mInboxCache(CStr(key))
                mInboxDirty(CStr(key)) = False
            End If
        End If
    Next key
End Sub
```


### Layer 3 — Background sync (Option B sync side)

```vba
Public Sub SyncStagingToNasInbox()
    ' Called by: FlushInboxCache, OnTime timer, or processor pre-run hook
    Dim stagingFiles As Variant
    Dim i As Long

    stagingFiles = ResolvePendingStagingFilesRole()
    If IsEmpty(stagingFiles) Then Exit Sub

    For i = LBound(stagingFiles) To UBound(stagingFiles)
        MergeOneStagingFileToNas CStr(stagingFiles(i))
    Next i
End Sub

Private Sub MergeOneStagingFileToNas(ByVal stagingPath As String)
    ' 1. Read all lines from staging file
    ' 2. Open NAS inbox (via cache)
    ' 3. For each line: parse JSON, check EventID not already in inbox, append row
    ' 4. Delete or archive the staging file on success
    ' 5. Mark NAS inbox dirty → FlushInboxCache will save it
End Sub
```


### Wiring it into `QueueEventCore`

The revised write sequence becomes:

```vba
' Step 1: always write local staging (fast, offline-safe)
If Not AppendToLocalStagingInbox(eventType, resolvedWh, resolvedSt, rowDict, errorMessage) Then
    GoTo CleanExit   ' even local write failed — rare, surface the error
End If

' Return success to the caller immediately
QueueEventCore = True
eventIdOut = rowDict("EventID")

' Step 2: attempt NAS sync opportunistically (non-blocking if NAS is down)
On Error Resume Next
SyncStagingToNasInbox
On Error GoTo 0
```


## Failure Modes Covered

| Scenario | Outcome |
| :-- | :-- |
| NAS down during click | Local staging succeeds; click returns in ~5 ms; NAS sync retried on next click or timer |
| Excel crashes before NAS flush | Staging `.jsonl` survives on disk; sync runs on next session open |
| NAS inbox locked by processor | Local staging buffers; sync waits for lock release |
| Station offline for hours | All events accumulate in staging; bulk-merged when NAS reconnects |
| Duplicate sync (crash mid-merge) | EventID dedup check in `MergeOneStagingFileToNas` makes it idempotent |

## How This Shapes the R2 F\# Design

This two-layer model maps almost directly onto what R2 would do natively.  The local staging `.jsonl` file *becomes* the R2 inbox format — the F\# processor just reads it directly instead of requiring a merge step. The VBA merge layer (`SyncStagingToNasInbox`) effectively becomes a no-op in R2 because the processor reads from the same append-only file the writer produces. So implementing Option A+B in R1 VBA also pre-shapes the on-disk format toward what R2 expects, making the migration cheaper.[^10_1]

# What do you think of this adaptation to A+B plan?:

    1. On Make/Unmake click:
    	○ Validate auth/context.
    	○ Build the normal inbox row.
    	○ Append it to local staging JSONL.
    	○ Return success immediately: “Queued locally; NAS sync pending.”
    	○ Do not touch the NAS inbox in the click path.
    2. Background/on-demand sync:
    	○ Merge local staging rows into the NAS .xlsb inbox.
    	○ Use EventID dedupe.
    	○ Save once per batch.
    	○ Archive/delete staged rows only after successful NAS save.
    3. Processor safety:
    	○ Add a processor pre-run hook to call staging sync first.
    	○ Add a timer or “Sync Now” command for normal operation.
    	○ Form sync message should distinguish Local queued, NAS synced, and Processed.
    This is a solid plan. The sequencing is correct and the three-layer status model is the right UX call. A few thoughts on each part:

## Part 1 — Click Path

The design is right. One thing worth locking down explicitly: **auth and context validation must still happen synchronously in the click path**, before the local append. The temptation later will be to defer auth too ("just queue it and validate on sync") — don't. If the user's session is expired or they lack `SHIP_POST`, they need to know *now*, not when the sync runs minutes later with no UI context to surface the error. The local staging file should only ever contain pre-validated events.

Also worth deciding now: **does the local staging file survive an Excel restart?** It should — `LOCALAPPDATA\invSys\staging\` persists across sessions, which is the whole point of resilience. That means `SyncStagingToNasInbox` should also be called on **add-in startup** (in `App_WorkbookOpen` or equivalent), not just on timer and pre-processor-run. Otherwise a crash between click and sync leaves events stranded until the user manually triggers something.

## Part 2 — Sync

The design is correct. The one implementation detail that matters most: **the staging file must not be deleted until after `wb.Save` returns without error**. The sequence must be strictly:

```
1. Open / get cached NAS inbox
2. Read staging lines
3. Dedup against existing EventIDs in NAS table
4. Append new rows to NAS table
5. wb.Save   ← must succeed
6. Archive/delete staging file
```

If you delete the staging file before step 5, a save failure loses events permanently. If you archive first (rename to `.synced`) and only delete `.synced` files on the *next* successful sync cycle, you get a two-phase commit without needing transactions.

One addition worth making: **the sync should be a mutex/flag-guarded operation** — if a sync is already in progress (timer fired while a previous sync is still running), skip rather than stack. A simple module-level `mSyncInProgress As Boolean` is sufficient for single-threaded VBA.

## Part 3 — Processor Safety

The pre-run hook is essential and correctly placed. One nuance: the processor hook should **wait for sync completion** before opening the NAS inbox, not fire-and-forget. In VBA that's naturally synchronous anyway, but worth stating explicitly so no future refactor makes it async and breaks the ordering guarantee.

The three-status model — **Local queued → NAS synced → Processed** — is the right distinction. A suggested mapping for the form message:


| State | Source of truth | Message |
| :-- | :-- | :-- |
| Local queued | Staging `.jsonl` exists, not yet in NAS inbox | `⏳ Queued locally — sync pending` |
| NAS synced | Row in NAS inbox with `Status = NEW` | `☁ Synced to NAS — awaiting processor` |
| Processed | Row in NAS inbox with `Status = PROCESSED` | `✓ Posted` |
| Poison | Row with `Status = POISON` | `⚠ Post failed — [ErrorCode]` |

The form can derive this by checking: does a staging line exist for this EventID? If no, is the row in the NAS inbox? If yes, what is its Status?

## One Gap Worth Flagging

The plan doesn't yet address **what happens if the NAS inbox is locked by the processor during sync**. The current `ResolveInboxWorkbookForEventType` returns an error if the workbook is read-only.  The sync path should treat a locked NAS inbox as a *retry later* condition rather than a failure — the staging file stays intact and sync retries on the next timer tick. That's already implied by "archive only after successful save" but worth making explicit in the retry logic so the lock contention window between writer and processor is handled gracefully rather than surfacing an error to the user.

Overall the plan is well-structured and implementable as described. The only real additions are: startup sync on add-in open, two-phase archive (rename before delete), mutex guard on the sync function, and explicit lock-contention retry handling.

Codex has it right. The design is well-reasoned and the constraints are the right ones. Here's where it's fully correct, where it could be tightened, and one important thing it didn't say:

## Where Codex Is Correct

**Event-based repair is the right architecture.** Writing an `ADMIN_SHIPMENT_RECONCILE` event that the processor applies — rather than directly patching `NAS Inv` — keeps the audit log contiguous. Any future replay of `tblInventoryLog` will include the correction as a first-class record with a reason and an admin principal, not a mysterious jump in qty. This is the same reason `RECEIVE` and `SHIP` events exist: so inventory is always *derived* from the event stream, never directly mutated.

**Blocking repair when a live lock explains the discrepancy is critical.** If `Locked > 0` and there is an active shipment row still in flight, the difference between `NAS Inv` and `Projected Inv` is intentional. An admin tool that adjusts through an active lock would corrupt the reservation ledger. Codex's "inspect before adjusting" rule for `T27 v1/v2` is exactly right.

**"Preview only" first pass is the right default UX.** The first interaction should never write anything. It should render the diagnostic table and the proposed delta, then require an explicit second confirmation to queue the event.

***

## What to Tighten

**The event payload needs one more field: the reconciliation evidence source.** The `ADMIN_SHIPMENT_RECONCILE` event should carry the `EventId` of the `Shipments Sent` event it is correcting, not just a free-text reason. This lets the processor and the log viewer show "this admin adjustment corrects ShipmentsSent event `EVT-2026-0618-T27-001`" rather than relying on the admin remembering to type a coherent reason string. Without that link, the correction is an orphan in the log.

**The mismatch flags need a fourth case.** Codex lists three (`Projected < NAS` with no active row, `Locked > 0` with no active row, completed reservation still counted active). The scenario the screenshot showed adds a fourth: **`NAS Inv` increased after `Shipments Sent`**, meaning the post-sent reload overwrote the deducted value with a stale snapshot. The flag for this is: `NAS Inv` *after* the most recent `SHIP` event timestamp is higher than `NAS Inv` at the moment the event was written. The tool needs to compare the current value against the event log's last `SHIP` record, not just compare `NAS` vs `Projected` at rest.

**Require a human-readable "repair narrative" in the confirmation step.** Before writing the event, force the admin to type a one-line reason (not a dropdown, a free text field). This prevents the tool from being used as a lazy sync button with a boilerplate reason like "reconcile". The processor can store it in `tblInventoryLog.Notes`.

***

## The One Thing Codex Didn't Say

**This tool will expose whether the processor itself is idempotent on admin events.** The first time you write and replay an `ADMIN_SHIPMENT_RECONCILE` event you'll immediately find out if the processor handles it correctly or double-applies it on re-load. Build the processor handler for this event type *before* building the UI, write a unit test that applies the event twice and asserts the qty only changed once, and only then wire up the admin form. If you do it the other way (build the UI first), you'll ship a reconciliation tool that can accidentally reconcile twice.

That's the only gap. Otherwise, build it exactly as Codex described.
